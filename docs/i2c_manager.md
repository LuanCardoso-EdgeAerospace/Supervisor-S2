# I2C Manager Module

## Purpose

`i2c_manager` serializes access to the STM32 HAL I2C memory API from multiple FreeRTOS tasks.

The module exposes blocking functions with a HAL-like interface:

- `I2C_Mem_Write(...)`
- `I2C_Mem_Read(...)`

Each call posts an I2C request to a CMSIS-RTOS2 message queue assigned to the selected physical bus. A dedicated manager task for that bus receives the request, performs the actual `HAL_I2C_Mem_Write()` or `HAL_I2C_Mem_Read()`, stores the result, and releases a semaphore so the caller can continue.

The caller does not access the HAL I2C peripheral directly. Only the manager task touches the selected `I2C_HandleTypeDef`.

---

## Problem this module solves

The STM32 HAL I2C blocking API is not safe to call concurrently from several RTOS tasks against the same peripheral handle.

Typical failure modes when several tasks call `HAL_I2C_Mem_Read()` or `HAL_I2C_Mem_Write()` directly include:

- overlapping transactions on the same bus
- HAL state conflicts, usually around `hi2c->State`
- timeout behavior depending on task scheduling
- bus recovery issues becoming harder to isolate
- inconsistent device behavior when two clients assume exclusive bus ownership

This module gives each I2C bus a single execution context. Multiple tasks may request transfers, but the physical bus is used by one manager task in FIFO queue order.

## Current design

### One queue per bus

The module supports three I2C buses:

```c
typedef enum {
    I2C_BUS_1 = 0,
    I2C_BUS_2,
    I2C_BUS_3,
    I2C_BUS_COUNT
} I2C_BusId_t;
```

Each bus is associated with one CMSIS-RTOS2 message queue:

```c
extern osMessageQueueId_t i2c1QueueHandle;
extern osMessageQueueId_t i2c2QueueHandle;
extern osMessageQueueId_t i2c3QueueHandle;
```

These queue handles are expected to be generated or declared by the application, normally in `app_freertos.c` when using CubeMX.

### One manager task per bus

Each bus has one task entry point:

```c
void I2C1_ManagerTask(void *argument);
void I2C2_ManagerTask(void *argument);
void I2C3_ManagerTask(void *argument);
```

The task entry points all call the same internal task body with a fixed bus ID.

The manager task blocks on its bus queue. When a request arrives, it performs the corresponding HAL operation and signals the waiting caller.

### Direct use of application-level I2C handles

The module does not require an explicit initialization function.

It uses the I2C handles generated by CubeMX and declared in `main.c`:

```c
extern I2C_HandleTypeDef hi2c1;
extern I2C_HandleTypeDef hi2c2;
extern I2C_HandleTypeDef hi2c3;
```

The bus ID selects the matching handle internally.

This avoids having to call an extra `I2C_Manager_Init()` function during startup. The module depends on the same global handles already used by the rest of the STM32 project.

## Request format

Requests are represented by `I2C_Request_t`:

```c
typedef struct {
    I2C_OpType_t    op;
    uint16_t        dev_addr;
    uint16_t        mem_addr;
    uint16_t        mem_addr_size;
    uint16_t        size;
    uint8_t         data[I2C_MGR_MAX_DATA_LEN];
    HAL_StatusTypeDef result;
    osSemaphoreId_t done_sem;
} I2C_Request_t;
```

The request contains everything the manager task needs to execute the transfer.

For writes, the caller data is copied into `data[]` before the request is queued.

For reads, the manager task fills `data[]`, and the caller receives a copy after the transaction completes successfully.

The payload size is limited by:

```c
#define I2C_MGR_MAX_DATA_LEN 64U
```

Increase this value if a device needs larger transfers. This increases the size of every queued request, so adjust the queue memory accordingly.

## Public API

### `I2C_Mem_Write`

```c
HAL_StatusTypeDef I2C_Mem_Write(I2C_BusId_t bus,
                                uint16_t DevAddress,
                                uint16_t MemAddress,
                                uint16_t MemAddSize,
                                uint8_t *pData,
                                uint16_t Size,
                                uint32_t timeout_ms);
```

Queues a memory write operation on the selected I2C bus.

Parameters match `HAL_I2C_Mem_Write()` except for the first parameter and the timeout meaning:

- `bus`: selected physical I2C bus
- `DevAddress`: HAL-formatted device address, normally 7-bit address shifted left by one
- `MemAddress`: target register or memory address inside the device
- `MemAddSize`: `I2C_MEMADD_SIZE_8BIT` or `I2C_MEMADD_SIZE_16BIT`
- `pData`: pointer to source data
- `Size`: number of bytes to write
- `timeout_ms`: maximum time the caller waits to enqueue and complete the request

Returns:

- `HAL_OK`: operation completed successfully
- `HAL_ERROR`: invalid parameters, missing queue, or semaphore allocation failure
- `HAL_TIMEOUT`: request could not be queued or completed before `timeout_ms`
- other HAL status values may be propagated from `HAL_I2C_Mem_Write()` if returned by HAL

### `I2C_Mem_Read`

```c
HAL_StatusTypeDef I2C_Mem_Read(I2C_BusId_t bus,
                               uint16_t DevAddress,
                               uint16_t MemAddress,
                               uint16_t MemAddSize,
                               uint8_t *pData,
                               uint16_t Size,
                               uint32_t timeout_ms);
```

Queues a memory read operation on the selected I2C bus.

Parameters match `HAL_I2C_Mem_Read()` except for the first parameter and the timeout meaning.

On success, the received bytes are copied into `pData`.

Returns the same status classes as `I2C_Mem_Write()`.

## Timeout behavior

`timeout_ms` is a caller-side timeout. It is used for two waits:

1. waiting for space in the message queue
2. waiting for the manager task to signal completion

Inside the manager task, the HAL call currently uses `HAL_MAX_DELAY`:

```c
HAL_I2C_Mem_Write(..., HAL_MAX_DELAY);
HAL_I2C_Mem_Read(..., HAL_MAX_DELAY);
```

This means the caller timeout does not directly limit the HAL transaction once the manager task has started it.

Implication: if the HAL driver blocks indefinitely because the bus is stuck, the manager task can remain blocked. Later requests on the same bus will not execute until the HAL call returns or the system recovers the bus.

This is intentional for the current version because the module separates request scheduling from low-level bus recovery. If strict bus-level deadlines are required, pass a finite HAL timeout inside `i2c_task_run()` instead of `HAL_MAX_DELAY`.

## Design decisions

### Dedicated manager task instead of a mutex only

A mutex around direct HAL calls would prevent concurrent access, but every client task would still execute I2C transactions in its own context.

Using a manager task gives stricter ownership:

- only one task calls HAL for a given I2C peripheral
- bus access order is centralized
- all request paths use the same timeout and result handling
- future bus recovery can be placed in one location
- driver code for sensors and peripherals does not need to manage I2C locking directly

This is heavier than a mutex, but it is more predictable for a system with several tasks and several I2C devices.

### Queue request copied by value

The request is posted to the queue by value.

This avoids keeping a pointer to a caller-owned stack object inside the queue. The caller blocks until completion, but the queue still owns a complete copy of the request descriptor. This reduces lifetime mistakes.

The semaphore handle inside the request is still created by the caller and copied into the queued request. The manager task releases that semaphore when the HAL transaction finishes.

### Per-request semaphore

Each request gets its own binary semaphore.

This avoids needing one global completion object per bus or per caller. The manager task does not need to know which task submitted the request. It only releases the semaphore stored in the request.

This design costs one semaphore allocation per transaction. For high-rate traffic, a future optimization could use static semaphores, a small request pool, or a client-owned completion object.

### Fixed payload buffer

The request contains a fixed `data[]` array instead of a pointer to caller memory.

Reasons:

- write requests remain valid even if the caller buffer changes while waiting
- read data is returned only after successful completion
- queue items are self-contained
- the manager task never depends on external buffer lifetime

The tradeoff is a fixed maximum transfer size and larger queue item size.

### Extern CubeMX handles instead of `I2C_Manager_Init()`

The current module directly references these application handles:

```c
hi2c1
hi2c2
hi2c3
i2c1QueueHandle
i2c2QueueHandle
i2c3QueueHandle
```

This matches the generated STM32 project structure and removes an extra initialization step.

The previous design required the application to pass handles into the module. That created another startup dependency and another possible failure mode. In this design, if CubeMX generated the handles and the names are unchanged, the module can use them directly.

The cost is tighter coupling to handle names. Porting to a project with different names requires changing the extern declarations and the switch functions.

## CubeMX / FreeRTOS setup

### 1. Enable the I2C peripherals

Enable the I2C instances used by the project. For the current module, the expected HAL handles are:

```c
hi2c1
hi2c2
hi2c3
```

If only one or two buses are used, unused task entry points may remain unused. The enum and switch logic can also be reduced.

### 2. Create one message queue per bus

Create CMSIS-RTOS2 message queues with these handle names:

```c
i2c1QueueHandle
i2c2QueueHandle
i2c3QueueHandle
```

The queue item size must be:

```c
sizeof(I2C_Request_t)
```

The queue depth depends on expected traffic. A depth of 4 to 8 is usually enough for low-rate sensor traffic. Use a larger queue only if several tasks may submit bursts.

### 3. Create one manager task per active bus

Assign the task entry point for each bus:

```c
I2C1_ManagerTask
I2C2_ManagerTask
I2C3_ManagerTask
```

Each manager task should have enough stack for the HAL call path and local request object. Start conservatively and verify with stack high-water marks.

### 4. Include the module

Add the following files to the build:

```text
i2c_manager.c
i2c_manager.h
```

Include the header in device drivers or tasks that need managed I2C access:

```c
#include "i2c_manager.h"
```

No explicit module initialization call is required.

## Usage examples

### Write one register

```c
#include "i2c_manager.h"

#define TEMP_SENSOR_ADDR   (0x48U << 1)
#define CONFIG_REG         0x01U

void temp_sensor_configure(void)
{
    uint8_t config = 0x60U;

    HAL_StatusTypeDef st = I2C_Mem_Write(I2C_BUS_1,
                                         TEMP_SENSOR_ADDR,
                                         CONFIG_REG,
                                         I2C_MEMADD_SIZE_8BIT,
                                         &config,
                                         1U,
                                         100U);

    if (st != HAL_OK) {
        /* Handle error */
    }
}
```

### Read one register

```c
#include "i2c_manager.h"

#define TEMP_SENSOR_ADDR   (0x48U << 1)
#define STATUS_REG         0x00U

HAL_StatusTypeDef temp_sensor_read_status(uint8_t *status)
{
    return I2C_Mem_Read(I2C_BUS_1,
                        TEMP_SENSOR_ADDR,
                        STATUS_REG,
                        I2C_MEMADD_SIZE_8BIT,
                        status,
                        1U,
                        100U);
}
```

### Read a multi-byte value

```c
#include "i2c_manager.h"

#define IMU_ADDR           (0x68U << 1)
#define ACCEL_XOUT_H       0x3BU

HAL_StatusTypeDef imu_read_accel_x(int16_t *accel_x)
{
    uint8_t raw[2];

    HAL_StatusTypeDef st = I2C_Mem_Read(I2C_BUS_2,
                                        IMU_ADDR,
                                        ACCEL_XOUT_H,
                                        I2C_MEMADD_SIZE_8BIT,
                                        raw,
                                        sizeof(raw),
                                        100U);
    if (st != HAL_OK) {
        return st;
    }

    *accel_x = (int16_t)((raw[0] << 8) | raw[1]);
    return HAL_OK;
}
```

### Write a 16-bit register address

```c
#include "i2c_manager.h"

#define EEPROM_ADDR        (0x50U << 1)
#define EEPROM_WORD_ADDR   0x0120U

HAL_StatusTypeDef eeprom_write_byte(uint8_t value)
{
    return I2C_Mem_Write(I2C_BUS_3,
                         EEPROM_ADDR,
                         EEPROM_WORD_ADDR,
                         I2C_MEMADD_SIZE_16BIT,
                         &value,
                         1U,
                         100U);
}
```

## Porting guide

### Porting to a project with different I2C handle names

If the project does not use `hi2c1`, `hi2c2`, or `hi2c3`, update the extern declarations in `i2c_manager.c`:

```c
extern I2C_HandleTypeDef hi2c1;
extern I2C_HandleTypeDef hi2c2;
extern I2C_HandleTypeDef hi2c3;
```

Then update `i2c_get_handle()` so each `I2C_BUS_x` maps to the correct handle.

Example:

```c
extern I2C_HandleTypeDef hi2c4;

static I2C_HandleTypeDef *i2c_get_handle(I2C_BusId_t bus)
{
    switch (bus) {
    case I2C_BUS_1: return &hi2c4;
    default:        return NULL;
    }
}
```

### Porting to fewer buses

For a single-bus project:

1. Keep only the required enum value.
2. Keep only the matching queue extern.
3. Keep only the matching manager task entry point.
4. Remove unused switch cases.

Minimal enum example:

```c
typedef enum {
    I2C_BUS_1 = 0,
    I2C_BUS_COUNT
} I2C_BusId_t;
```

### Porting to more buses

To add another bus:

1. Add another enum value before `I2C_BUS_COUNT`.
2. Add the matching `extern I2C_HandleTypeDef` declaration.
3. Add the matching queue handle extern.
4. Add cases in `i2c_get_handle()` and `i2c_get_queue()`.
5. Add a task entry point that calls `i2c_task_run()` with the new bus ID.
6. Create the queue and task in CubeMX or application startup code.

Example additions:

```c
typedef enum {
    I2C_BUS_1 = 0,
    I2C_BUS_2,
    I2C_BUS_3,
    I2C_BUS_4,
    I2C_BUS_COUNT
} I2C_BusId_t;
```

```c
extern I2C_HandleTypeDef hi2c4;
extern osMessageQueueId_t i2c4QueueHandle;
```

```c
void I2C4_ManagerTask(void *argument)
{
    (void)argument;
    i2c_task_run(I2C_BUS_4);
}
```

### Porting to a non-CubeMX project

The module does not require CubeMX, but it does require these symbols to exist:

```c
I2C_HandleTypeDef hi2c1;
osMessageQueueId_t i2c1QueueHandle;
```

For a manual project, define the I2C handles and queue handles in one C file and keep the extern declarations in `i2c_manager.c`.

The queue must be created before any task calls `I2C_Mem_Read()` or `I2C_Mem_Write()`.

### Porting to native FreeRTOS instead of CMSIS-RTOS2

The current code uses CMSIS-RTOS2 APIs:

- `osMessageQueuePut()`
- `osMessageQueueGet()`
- `osSemaphoreNew()`
- `osSemaphoreAcquire()`
- `osSemaphoreRelease()`
- `osSemaphoreDelete()`
- `osDelay()`

For native FreeRTOS, replace these with equivalent queue and semaphore calls:

- `xQueueSend()`
- `xQueueReceive()`
- `xSemaphoreCreateBinary()`
- `xSemaphoreTake()`
- `xSemaphoreGive()`
- `vSemaphoreDelete()`
- `vTaskDelay()`

Also replace the queue and semaphore handle types in `I2C_Request_t`.

## Integration notes

### Device address format

`DevAddress` must use the STM32 HAL format.

For normal 7-bit I2C addresses, shift the address left by one:

```c
#define DEVICE_ADDR (0x48U << 1)
```

Do not pass the unshifted 7-bit address unless the rest of the project intentionally wraps HAL differently.

### Maximum transfer length

The module rejects any transfer larger than `I2C_MGR_MAX_DATA_LEN`.

```c
if (Size > I2C_MGR_MAX_DATA_LEN || pData == NULL) {
    return HAL_ERROR;
}
```

For larger devices such as EEPROMs, split transfers into chunks or increase `I2C_MGR_MAX_DATA_LEN`.

### Do not call from interrupts

The public API allocates a semaphore, posts to a queue, and waits for completion. It is intended for task context only.

Do not call `I2C_Mem_Read()` or `I2C_Mem_Write()` from an ISR.

### Avoid direct HAL access to managed buses

Once a bus is managed by this module, all task-level access to that bus should go through the manager.

Calling `HAL_I2C_Mem_Read()` or `HAL_I2C_Mem_Write()` directly from another task bypasses the queue and can reintroduce concurrency bugs.

### Queue depth and priority

The manager task priority affects I2C latency.

Use a priority high enough that normal clients are not blocked longer than expected, but avoid making the task higher than critical timing tasks unless the I2C devices require it.

The queue depth controls how many pending requests can wait before callers start timing out while posting.

## Limitations

- Only memory-style HAL transfers are currently wrapped.
- Plain transmit/receive operations are not implemented.
- DMA and interrupt-driven HAL transfers are not implemented.
- The HAL transaction timeout is currently `HAL_MAX_DELAY` inside the manager task.
- Request payload size is fixed at compile time.
- The module is coupled to specific global handle names unless ported.
- The public API is blocking from the caller perspective.

## Possible future changes

### Add plain transmit/receive wrappers

Some devices do not use register-addressed memory operations. For those devices, add request types for:

```c
HAL_I2C_Master_Transmit()
HAL_I2C_Master_Receive()
```

This requires extending `I2C_OpType_t` and the task switch logic.

### Add finite HAL transaction timeout

Replace `HAL_MAX_DELAY` inside `i2c_task_run()` with a bounded timeout.

This makes bus failures return to the manager task instead of potentially blocking the task forever.

### Add bus recovery

If a transaction fails or times out, the manager task is the correct place to centralize recovery.

Possible recovery actions include:

- deinitializing and reinitializing the HAL I2C peripheral
- clocking SCL manually to release a stuck slave
- resetting the peripheral
- recording error counters per bus

### Use static synchronization objects

The current design allocates and deletes a semaphore for each transaction.

For systems with high I2C traffic, replace dynamic semaphore allocation with one of these:

- caller-owned completion object
- static semaphore pool
- fixed request pool
- task notification directed to the submitting task

This reduces heap usage and allocation overhead.

## Expected application ownership

The application owns:

- I2C peripheral configuration
- queue creation
- manager task creation
- bus recovery policy
- device-specific drivers

The module owns:

- selecting the correct queue for a bus
- selecting the correct HAL handle for a bus
- serializing requests
- copying request payloads
- calling the HAL I2C memory API
- returning the final HAL status to the caller
